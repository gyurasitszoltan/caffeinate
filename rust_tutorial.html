Rust kezdőknek — a KeepAwake projekten keresztül

Egy lépésről lépésre haladó tutorial, ami nulláról indul, és egy valós macOS menüsor-alkalmazás (egy kávéscsésze ikonnal ébren tartott gép) forráskódját magyarázza el kódonként.

Rust 2021 edition macOS / IOKit FFI tao + tray-icon

1.Mi az a Rust?

A Rust egy modern, rendszerprogramozási nyelv, amit a Mozilla indított 2010-ben. Két fő célja van:

  • Teljesítmény — olyan gyors, mint a C/C++, nincs garbage collector (szemétgyűjtő), hanem a memóriát fordítás időben, úgynevezett ownership (birtoklás) szabályokkal kezeli.
  • Biztonság — a fordító (a borrow checker) már fordítás időben kiszűri a memóriaszivárgást, a data race-eket (párhuzamossági hibákat), és a null-mutató hozzáférést.

Ez azt jelenti, hogy ha a Rust programod lefordul, abban bizonyos osztályú hibák biztosan nincsenek. Ezért szeretik biztonságkritikus rendszereknél, CLI eszközöknél, és most mi is egy ilyen nyelven írunk egy kis asztali appot.

Miért pont ez a projekt? A KeepAwake érinti a Rust legtöbb fontos témáját: enum-ok és mintaillesztés (pattern matching), Result/Option hibakezelés, trait-ek (pl. Drop, Default), modulrendszer, külső függőségek, FFI (külső C könyvtár hívása), lifecycle-kezelés, és még unit tesztek is. Célszerű tananyag.

3.Hello, Rust!

Mielőtt a KeepAwake-be beleásnánk magunkat, lássuk a legegyszerűbb Rust programot. Egy új main.rs fájl tartalma:

fn main() {
    println!("Hello, Rust!");
}

Vegyük sorra — ez a 3 sor rengeteget elárul a nyelvről:

  • fn main() — a fn kulcsszó függvényt vezet be. A main a belépési pont: a futtatható program innen indul, mint a C-ben.
  • println! — a ! végén jelzi, hogy ez nem függvény, hanem egy makró. A makrók fordítás előtt kiegészítik a kódot. A println! helyére a fordító kódot generál, ami a szöveget kiírja a standard outputra.
  • A "Hello, Rust!" egy &'static str típusú string literál.
  • A sor végén a ; kötelező — a Rust utasításokat zárja.
Miért makró a println? Mert változó számú argumentumot vesz át: println!("{} éve", 10). A Rust makrók fordítási időben kiegészülnek a megfelelő kóddá. Ez gyorsabb és biztonságosabb, mint a C printf futási idejű formátum-ellenőrzése.

4.A KeepAwake project bemutatása

A KeepAwake egy macOS menüsor (tray) alkalmazás. Célja: egy kattintással megakadályozni, hogy a Mac elaludjon — például mikor letöltünk valamit, vagy fut egy hosszú build. A menüsorban egy kávéscsésze ikon mutatja a hátralévő időt: tele csésze = sok idő, üres csésze = lejárt / inaktív.

Ezt úgy éri el, hogy a macOS IOKit rendszer-API-jával létrehoz egy power assertion-t, ami jelzi a rendszernek: "ne engedd a gépet aludni". Amikor leállítjuk (vagy kilépünk), ezt az assertion-t release-eljük.

Az architektúra

main.rs ─ tao event loop, másodperces tick, UI frissítés │ ├── tray.rs ─ menü felépítés + ID -> parancs ├── app_state.rs ─ alkalmazás állapot modell (Inactive/Indefinite/Timed/Expired) ├── timer.rs ─ időzített mód állapota (hátralévő/eltelt idő) ├── icons.rs ─ PNG frame cache + frame kiválasztás hátralévőből └── awake.rs ─ IOKit FFI + assertion életciklus (start/stop/Drop)

Minden modul egy-egy felelősséget kapott. Ezt a szétválasztást a Rust modulrendszere teszi lehetővé. Most pedig építsük fel lépésről lépésre!

A Cargo minden projektjének van egy Cargo.toml manifest-fájlja a gyökérben. Ez mondja meg, mi a projekt neve, milyen függőségei vannak, stb. A KeepAwake-é:

TOML magyarázat A TOML egy konfigurációs formátum, ami szekciókra ([package], [dependencies]) van osztva. A "0.35" verziószámok a semver (szemantikus verziózás) szerinti tartományt jelentenek: legalább 0.35, de kisebb mint 0.36. A Cargo ezeket automatikusan letölti és lefordítja.

6.A Rust modulrendszer

A src/main.rs tetején ez az 5 sor szerepel — ez vezeti be a többit:

src/main.rs (eleje)
mod app_state;
mod awake;
mod icons;
mod timer;
mod tray;

A mod foo; utasítás azt mondja a fordítónak: "van egy foo nevű modulom, keresd a src/foo.rs fájlban". A modulokkal egy nagy fájlt több kisebbre bonthatunk úgy, hogy minden fájl egy-egy felelősségért feleljen.

  • Modul = egy névtér név-és-kód számára. A fájlnevek és modulnevek összekapcsolódnak.
  • Láthatóság: egy modulban pub fn / pub struct a publikus, a többi privát. Csak a pub-lik látszanak kifelé.
  • Útvonal: app_state::AppState jelentése: "az app_state modulban lévő AppState típus". Ez a :: útvonal-szeparátor.
Privát alapértelmezés A Rustban minden privát, ha nem mondod a fordítónak az ellenkezőjét. Ez biztonságos alapbeállítás: nem szivároghat ki belső részlet. Ha egy másik modulból el akarod érni, pub kulcsszót kell tenni elé.

A fájl tetején lévő nagy //! ... blokk egy belső dokumentációs komment: a fordító a modulhoz (fájlhoz) rendeli, nem egy konkrét elemhez. Ezekből a cargo doc HTML dokumentációt tud generálni.

7.Lépés 1 — Állapotmodell (app_state.rs)

Kezdjük a legegyszerűbb modullal. Ez a fájl tartalmazza a teljes alkalmazás állapotát — minden UI frissítés ebből származik. Itt találkozunk először a Rust legfontosabb eszközével: az enum-mal.

src/app_state.rs
pub enum AwakeMode {
    Inactive,
    Indefinite,
    Timed(TimerState),
    Expired,
}

Egy enum egy diszjunkt unió: egy érték pontosan egy variáns lehet egyszerre. Ez hasonlít a C tagged union-jára vagy a TypeScript discriminated union-jére, de sokkal biztonságosabb. Itt az ébren tartás 4 lehetséges módját soroljuk fel:

  • Inactive — semmi sem blokkolva
  • Indefinite — manuális, időkorlát nélküli ébren tartás
  • Timed(TimerState) — időzített mód, és a variáns adatot is hordoz!
  • Expired — a timer lejárt (tranziens nyugalmi állapot)
Eltérés a többi nyelvtől Java/C#-ban ehhez osztályhierarchiát (class Indefinite extends AwakeMode) kellene írni. A Rust enum variánsai hordozhatnak tetszőleges adatot, így sokkal tömörebb. A mintaillesztés (lent) kikényszeríti, hogy minden variánst kezelj — nem felejthetsz el egyet sem.

Mintaillesztés (pattern matching)

Az enum értékeket a match utasítással vizsgáljuk meg:

pub fn is_active(&self) -> bool {
    matches!(self, AwakeMode::Indefinite | AwakeMode::Timed(_))
}

A matches! makró egy rövidítése a match-nek: true-t ad, ha az érték beleillik a mintába. A _ jelentése "bármi" — a Timed(_) minta illeszkedik bármilyen Timed variánsra, az adatot nem vizsgálja. Tehát aktív a session, ha vagy Indefinite, vagy Timed bármilyen timer-rel.

Egy struct és impl

pub struct AppState {
    pub awake_mode: AwakeMode,
    pub keep_display_awake: bool,
    pub current_icon_frame: IconFrame,
}

impl AppState {
    pub fn new() -> Self {
        Self {
            awake_mode: AwakeMode::Inactive,
            keep_display_awake: false,
            current_icon_frame: IconFrame::Empty,
        }
    }
    // ...további metódusok
}
  • struct = adatmezők halmaza (mint a C struct, de metódusokkal bővíthető).
  • impl AppState { ... } = az AppState típushoz tartozó metódusok blokkja. Ide kerülnek a függvények.
  • Self = a típus saját nevének rövidítése az impl blokkon belül.
  • &self a metódus első paramétere = mint a C++ this, csak a Rustban explicit kell írni, és a & jelzi, hogy kölcsönözve (borrow), nem birtokolva vesszük át.

A match ami adatot von ki

pub fn tooltip(&self) -> String {
    match &self.awake_mode {
        AwakeMode::Inactive => "KeepAwake: inaktív".to_string(),
        AwakeMode::Indefinite => "KeepAwake: aktív".to_string(),
        AwakeMode::Timed(t) => {
            let r = t.remaining();
            let m = r.as_secs() / 60;
            let s = r.as_secs() % 60;
            format!("KeepAwake: {:02}:{:02} van hátra", m, s)
        }
        AwakeMode::Expired => "KeepAwake: lejárt".to_string(),
    }
}

Itt látható a match igazi ereje: a Timed(t) minta nemcsak ellenőriz, hanem köt is — a t név a variánsban lévő TimerState-re hivatkozik, amit utána használhatunk. A match kikötelező: ha lehagysz egy variánst, a fordító hibát ad (így nem felejtessz el esetet). A format! makró a C printf-hez hasonlóan formáz, de típusbiztos és fordítási időben ellenőrzött.

A Default trait A fájl végén: impl Default for AppState { fn default() -> Self { Self::new() } }. A Default egy beépített trait (interfész), ami egy alapértelmezett értéket szolgáltat. Sok generikus API elvárja, így illik megadni.

8.Lépés 2 — Időzítő (timer.rs)

A timer.rs egy önálló modul, ami egy elindított időzítés adatait tárolja. Ez egy szép kis példa a Rust szabványos könyvtárára (std).

src/timer.rs
use std::time::{Duration, Instant};

pub struct TimerState {
    started_at: Instant,
    total: Duration,
}

impl TimerState {
    pub fn new(total: Duration) -> Self {
        Self {
            started_at: Instant::now(),
            total,
        }
    }

    pub fn elapsed(&self) -> Duration {
        Instant::now().duration_since(self.started_at)
    }

    pub fn remaining(&self) -> Duration {
        self.total.saturating_sub(self.elapsed())
    }

    pub fn is_finished(&self) -> bool {
        self.elapsed() >= self.total
    }
}
  • use std::time::{Duration, Instant}; — a std a szabványos könyvtár, a use behozza a neveket, hogy ne kelljen std::time::Duration-ként hivatkozni.
  • Instant::now() — egy monoton óra pillanatnyi időpontja. Ez a teljesítményméréshez jó (nem a faliórához!).
  • Duration — egy időtartam (másodperc + nanoszekundum).
  • saturating_sub — kivonás, ami 0-nál megáll, nem underflow-ol. Ez fontos Rust-trükk: a numerikus típusoknál alapból nincs overflow, de explicit kell mondani, mit szeretnénk tenni (saturating_*, wrapping_*, checked_*).

Egységtesztek

A fájl alján belső tesztmodul van — a Rust a teszteket a kóddal együtt, ugyanabban a fájlban tartja:

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn remaining_saturates_at_zero() {
        let t = TimerState::new(Duration::from_millis(10));
        std::thread::sleep(Duration::from_millis(30));
        assert!(t.is_finished());
        assert_eq!(t.remaining(), Duration::ZERO);
    }
}
  • #[cfg(test)] — attribútum, ami azt mondja: "ez a modul csak teszt buildben (cargo test) legyen lefordítva, a release binárisból kimarad".
  • use super::*; — "hozd be mindazt, amit a szülőmodulban publikálnak" (a super = szülőmodul).
  • #[test] — jelzi, hogy ez egy tesztfüggvény. A cargo test automatikusan megtalálja és lefuttatja.
  • assert! és assert_eq! — assertion-makrók; ha nem teljesül, a teszt elbukik részletes hibaüzenettel.
$ cargo test --lib timer running 2 tests test timer::tests::remaining_saturates_at_zero ... ok test timer::tests::progress_clamps_to_one ... ok test result: ok. 2 passed; 0 failed

9.Lépés 3 — Ikonok (icons.rs)

Ez a modul kezeli a kávéscsésze 5 PNG "frame"-ét, és eldönti, az idő múlásával éppen melyiket mutassa. Itt látni fogunk HashMap-et és enum másik arcát (repr-eket).

src/icons.rs
/* Az enum reprezentációja: egyszerű 0..=4 egész szám.
   Így könnyű belőle fájlnév-indexet csinálni. */
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub enum IconFrame {
    Empty,   // cup_0.png
    Low,     // cup_1.png
    Mid,     // cup_2.png
    High,    // cup_3.png
    Full,    // cup_4.png
}

Az #[derive(...)] attribútum a fordítót kéri meg, hogy automatikusan implementáljon bizonyos trait-eket. Itt:

  • Debug — {:?} formátumú kiíráshoz (debugolás)
  • Clone, Copy — érték-szerinti másolás (mert a változó csak 1 byte!)
  • PartialEq, Eq — == és != összehasonlítás
  • Hash — kulcs lehet egy HashMap-ben

Egy függvény: arány → frame

pub fn frame_for_remaining(remaining: Duration, total: Duration) -> IconFrame {
    if total.is_zero() {
        return IconFrame::Full;
    }
    let ratio = remaining.as_secs_f64() / total.as_secs_f64();
    if ratio > 0.80 { IconFrame::Full }
    else if ratio > 0.60 { IconFrame::High }
    else if ratio > 0.40 { IconFrame::Mid }
    else if ratio > 0.20 { IconFrame::Low }
    else { IconFrame::Empty }
}

Ez tipikus Rust-vezérlési szerkezet: a let ratio = ... egy változó bekötése (az érték a sor végén, típus bal oldalt elhagyható, mert a fordító kitalálja). Az if/else if/else lánc — utolsó ágnak kötelező else, hogy minden eset le legyen fedve (a kifejezéseknek értéke van!). A return a függvényből ugrik ki; e nélkül az utolsó kifejezés a visszatérési érték (nincs ; a végén!).

Konstans fordítási idejű útvonal

pub fn path_for_frame(frame: IconFrame) -> String {
    format!("{}/assets/cup_{}@2x.png", env!("CARGO_MANIFEST_DIR"), frame.index())
}

Az env!("CARGO_MANIFEST_DIR") egy fordítási idejű makró: a Cargo beállít egy környezeti változót a projekt gyökerével, és a env! ezt beégeti a binárisba. Így a futtatható állomány független attól, honnan indítod — mindig megtalálja az assets/ mappát a build helyétől. Ez nagyon hasznos trükk!

A cache: HashMap

use std::collections::HashMap;

pub struct IconCache {
    icons: HashMap<IconFrame, Icon>,
}

impl IconCache {
    pub fn new() -> Self {
        let mut icons = HashMap::with_capacity(5);
        for frame in IconFrame::all() {
            let path = path_for_frame(frame);
            let icon = load_icon(&path);
            icons.insert(frame, icon);
        }
        Self { icons }
    }
}
  • HashMap<K, V> — asszociatív tömb. Generikus típus, a <IconFrame, Icon> behelyettesíti a kulcs/érték típusokat.
  • let mut — a változó mutable (változtatható). A Rustban alapból minden immutable, mut kell a módosításhoz.
  • for x in iter — a ciklus. A IconFrame::all() egy [IconFrame; 5] tömböt ad vissza, ami iterálhatóvá válik.
  • &path — kölcsönzés: átadjuk a függvénynek, de nem adjuk oda a birtoklást, így path utána is használható.
Egy minta a fájlban: pánika hibára A load_icon .unwrap_or_else(|e| panic!(...)) használ. Ez azt jelenti: ha a PNG betöltése sikertelen, a program panikál (leáll). Ez itt szándékos: hiányzó ikon = kritikus hiba, értelmetlen lenne továbbmenni. Általában azonban a Result-t továbbadjuk (lást következő lépés!).

10.Lépés 4 — FFI az IOKit-hez (awake.rs)

Ez a legizgalmasabb modul. Itt hívunk natív C függvényeket a macOS IOKit framework-jéből. Ezt hívják FFI-nek (Foreign Function Interface).

C függvények deklarálása

type IOPMAssertionID = u32;
type IOReturn = i32;

#[link(name = "IOKit", kind = "framework")]
extern "C" {
    fn IOPMAssertionCreateWithName(
        assertion_type: *const c_void,
        assertion_level: u32,
        assertion_name: *const c_void,
        assertion_id: *mut IOPMAssertionID,
    ) -> IOReturn;

    fn IOPMAssertionRelease(assertion_id: IOPMAssertionID) -> IOReturn;
}
  • type T = U; — típus-álnév. Olvashatóbbá teszi a kódot (ahol u32 van, az valójában egy "IOKit ID").
  • #[link(name = "IOKit", ...)] — megmondja a linkernek, hogy a IOKit macOS framework-öt hozzá kell linkelni.
  • extern "C" — egy blokk, ami C-linkage-ű (C hívási konvenciójú) függvényeket deklarál. A törzs nincs megírva — a C könyvtárból jön.
  • *const c_void — nyers mutató (pointer), ami csak olvasható. A *mut írható mutató. A c_void a C void-ja.
unsafe világ A C függvényeket hívó kódot unsafe { ... } blokkba kell tenni. A Rust biztonsági garanciái csak a saját kódjára vonatkoznak — a C kódot nem tudja ellenőrizni, így a programozó felelőssége, hogy a hívás biztonságos legyen. Ez a Rust kompromisszuma: nem tilthatod a nem-biztonságos műveleteket (mint C# unsafe), de explicit jelezni kell őket — így a "hol lehet hiba" kör szűkül.

Az unsafe blokk

fn create_assertion(type_str: &str) -> Result<IOPMAssertionID, IOReturn> {
    let cf_type = CFString::new(type_str);
    let cf_name = CFString::new(ASSERTION_NAME);
    let mut id: IOPMAssertionID = 0;

    let result = unsafe {
        IOPMAssertionCreateWithName(
            cf_type.as_concrete_TypeRef() as *const c_void,
            ASSERTION_LEVEL_ON,
            cf_name.as_concrete_TypeRef() as *const c_void,
            &mut id,
        )
    };

    if result == IORETURN_SUCCESS { Ok(id) } else { Err(result) }
}

A Result típus

A Result<T, E> a Rust hibakezelésének alapja. Ez egy enum két variánssal:

enum Result<T, E> {
    Ok(T),   // siker, benne az eredmény
    Err(E),  // hiba, benne a hibaadat
}

A függvényünk itt Ok(id)-t ad sikeres IOKit hívás esetén (az új assertion ID-jával), vagy Err(ior)-t, ha az IOKit hibakódot adott vissza. A Result kényszeríti a hívót, hogy kezelje a hibát — nem felejtheti el. Ez a Rust filozófiája: a hiba egy érték, nem egy kivétel.

Option — a "van vagy nincs" típus

pub struct AwakeController {
    system_assertion_id: Option<IOPMAssertionID>,
    display_assertion_id: Option<IOPMAssertionID>,
}

Az Option<T> egy enum, Some(T) (van érték) vagy None (nincs). A Rust-ban nincs null: ha egy érték "lehet hiányzó", az Option. Így a None sosem keveredhetik értékkel, és a fordító rávesz, hogy ellenőrizd a Some-ot, mielőtt kiveszed belőle a tartalmat. Itt None = "nincs aktív assertion", Some(id) = "van, ez az ID".

Az idempotens start és a hibakezelés lánc

pub fn start(&mut self, keep_display_awake: bool) -> Result<(), AwakeError> {
    self.stop();  // előbb felszabadítjuk az esetleg meglévőt

    let sys_id = create_assertion(SYSTEM_TYPE).map_err(AwakeError::CreateSystem)?;
    self.system_assertion_id = Some(sys_id);

    if keep_display_awake {
        match create_assertion(DISPLAY_TYPE) {
            Ok(disp_id) => self.display_assertion_id = Some(disp_id),
            Err(code) => {
                self.stop();  // rollback
                return Err(AwakeError::CreateDisplay(code));
            }
        }
    }
    Ok(())
}
  • &mut self — mutable kölcsönzés: ez a metódus módosítja a struct mezőit, ezért kell a mut.
  • .map_err(...)? — a ? operátor! Ha Err, kilép a függvényből és továbbadja a hibát (felfelé terjed). Ha Ok, kicsomagolja a belső értéket. A .map_err előtte átalakítja a hibátípust.
  • Ok(()) — az üres eredmény ("sikeresen végeztem, semmit nem adok vissza"). Az () az unit típus, egyetlen értékkel, mint a C void.
  • A match + rollback-minta tranzakciós viselkedést ad: ha a display assertion létrehozása elbukik, a system assertion-t is visszagörgetjük.

A Drop trait — a Rust destruktorja

impl Drop for AwakeController {
    fn drop(&mut self) {
        self.stop();
    }
}

Ez a Rust egyik legszebb mechanizmusa. A Drop trait implementálásával megmondjuk: "amikor egy AwakeController érték kimegy a hatókörből (eltűnik a stack-ről vagy memóriából), automatikusan hívd meg a drop metódust". Ez garantálja, hogy az IOKit assertion-t mindig felszabadítjuk — akár normális kilépés, akár pánika, akár korai return. Ez a Rust RAII (Resource Acquisition Is Initialization) modellje, mint a C++ destruktora, de biztonságosabban és kikényszerítve.

Miért fontos ez? Ha elfelejtenénk stop-ot hívni kilépéskor, a gép "beragadna" ébren tartott módba, és a pmset -g assertions megmutatná a szivárgást. A Drop megakadályozza ezt: ha a processz kilép, az AwakeController megsemmisül, és a Drop::drop felszabadítja az assertion-t. Garancia nyelvi szinten!

11.Lépés 5 — A tray menü (tray.rs)

Ez a modul csak a UI-t ismeri: felépíti a menüt, és a menüpont ID-jét parancsra fordítja. Az IOKit logika nem él itt — ez a tiszta szétválasztás.

src/tray.rs
pub enum TrayCommand {
    StartIndefinite,
    StartTimed(Duration),
    Stop,
    ToggleDisplayAwake,
    Quit,
}

pub fn command_for_id(id: &MenuId) -> Option<TrayCommand> {
    let s = id.0.as_str();
    match s {
        "indefinite" => Some(TrayCommand::StartIndefinite),
        "t15" => Some(TrayCommand::StartTimed(Duration::from_secs(15 * 60))),
        "t30" => Some(TrayCommand::StartTimed(Duration::from_secs(30 * 60))),
        "t60" => Some(TrayCommand::StartTimed(Duration::from_secs(60 * 60))),
        "t120" => Some(TrayCommand::StartTimed(Duration::from_secs(2 * 60 * 60))),
        "stop" => Some(TrayCommand::Stop),
        "display" => Some(TrayCommand::ToggleDisplayAwake),
        "quit" => Some(TrayCommand::Quit),
        _ => None,  // minden más (separator, status) nem parancs
    }
}

A match itt sztring-literálokra illeszkedik. A _ a végén egy catch-all: bármilyen más string ide kerül. Ez kötelező a match-ben (kivéve ha már minden variáns le van fedve). A id.0 a tuple-struktúra első mezőjét éri el (a MenuId(pub String) struktúra egyetlen, névtelen mezője).

Menü felépítése

let status = MenuItem::with_id("status", "KeepAwake: inaktív", false, None);
let indefinite = MenuItem::with_id("indefinite", "Ébren tartás bekapcsolása", true, None);
let display = CheckMenuItem::with_id("display", "Kijelzőt is ébren tartsa", true, false, None);

let menu = Menu::new();
menu.append_items(&[
    &status,
    &PredefinedMenuItem::separator(),
    &indefinite,
    // ...t15, t30, t60, t120, stop
    &PredefinedMenuItem::separator(),
    &display,
    &PredefinedMenuItem::separator(),
    &quit,
]).expect("menü felépítés sikertelen");

Az append_items egy tömb referenciát (&[&MenuItem; N]) vár — minden egyes elem egy referencia (&), mert a tray-icon könyvtár a menüpontokra weak ref-ként hivatkozik (ezért kell a MenuHandles struct-ban élve tartani őket). A .expect(msg) az Option/Result unwrap-elése hibaüzenettel — ha None/Err, pánikál ezzel a szöveggel.

12.Lépés 6 — Az event loop (main.rs)

A main.rs köti össze mindent. Itt fut a tao event loop-ja, ami másodpercenként felébredve frissíti az ikont és a menü állapotát. Ez a fájl mutatja meg, hogyan "él együtt" az összes eddigi modul.

A belépési pont és az események

enum UserEvent {
    TrayIconEvent(tray_icon::TrayIconEvent),
    MenuEvent(tray_icon::menu::MenuEvent),
}

fn main() {
    let event_loop = EventLoopBuilder::<UserEvent>::with_user_event().build();

    // ... az eseményeket a loopba tereljük egy EventLoopProxy-n át
    let proxy = event_loop.create_proxy();
    tray_icon::menu::MenuEvent::set_event_handler(Some(move |event| {
        let _ = proxy.send_event(UserEvent::MenuEvent(event));
    }));

    event_loop.run(move |event, _window_target, control_flow| {
        // ... minden eseményt itt dolgozunk fel
    });
}
  • EventLoopBuilder<UserEvent> — generikus típusparaméterezés: az event loop-unk a saját UserEvent enumunkat fogadja a natív események mellett.
  • move |event| { ... } — egy closure (névtelen függvény, lambda). A move kulcsszó jelzi, hogy a closure birtokba veszi a külső változókat (itt proxy-t), nem csak kölcsönzi. Ez az ownership modell egy fontos momentuma!
  • let _ = proxy.send_event(...) — a let _ a visszatérési érték elvetése. Csendesen ignorálja az Err-t (ha a loop már leállt).

A loop magja: match az eseményeken

event_loop.run(move |event, _window_target, control_flow| {
    # Minden iterációban beállítjuk, mikor ébredjünk legközelebb:
    *control_flow = ControlFlow::WaitUntil(Instant::now() + TICK_INTERVAL);

    match event {
        Event::NewEvents(StartCause::Init) => {
            # Induláskor létrehozzuk a tray ikont
            tray_icon = Some(TrayIconBuilder::new()...build()?);
        }
        Event::UserEvent(UserEvent::MenuEvent(ev)) => {
            let cmd = command_for_id(&ev.id);
            match cmd {
                Some(TrayCommand::StartIndefinite) => {
                    awake.start(app_state.keep_display_awake)?;
                    app_state.awake_mode = AwakeMode::Indefinite;
                }
                // ...többi parancs
                None => {} # separator / ismeretlen
            }
        }
        Event::MainEventsCleared => {
            # Másodperces tick: timer lejárat + UI frissítés
        }
        Event::LoopDestroyed => { awake.stop(); }
        _ => {}
    }
});

Itt látható a teljes architektúra működése. Néhány megfigyelés:

  • A match event több szintű mintaillesztés: UserEvent(MenuEvent(ev)) egyszerre bontja le a külső és belső enumot, és köti ev-t.
  • A ? operátor itt is dolgozik — ha a start hibát ad, a closure-ból (és a programból) propagálódik a hiba. De vigyázat: main visszatérési típusa (), nem Result, így valójában egy korai return történik (a hiba elnyelődik ebben a mintában).
  • A MainEventsCleared ág a "másodperces frissítés" helye. Itt három lépés történik: (1) timer lejárat ellenőrzése, (2) ikon cseréje csak ha változott, (3) tooltip és menü állapot frissítése. Az utóbbi kettő cache-et használ (last_tooltip, last_stop_enabled), hogy csak valódi változáskor hívjon drága UI API-t. Ez a Rust "kölcsönzés-alapú" tervezés gyakori mintája.

Az "így frissítsünk csak ha kell" minta

let stop_enabled = app_state.is_active();
if Some(stop_enabled) != last_stop_enabled {
    last_stop_enabled = Some(stop_enabled);
    menu_handles.stop.set_enabled(stop_enabled);
}

Ez egy elegáns ötlet: last_stop_enabled egy Option<bool> — None jelenti, hogy még sosem frissítettünk (első iteráció). Csak akkor hívjuk meg set_enabled-et, ha tényleg megváltozott az érték. Ez csökkenti a felesleges UI hívásokat.

Miért ilyen bonyolult a main? Mert a GUI event loop modell. Itt nincs "várakozás" blokkoló hívás — a loop mindig fut, és az eseményeket dispatch-eli. A "tick" (másodperces frissítés) úgy valósul meg, hogy ControlFlow::WaitUntil-lal kérjük a következő felébredést. Ez a tao (cross-platform, eredetileg winit alapú) tipikus mintája.
